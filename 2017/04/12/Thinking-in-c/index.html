Thinking in c++

3月20号到4月5号，稀稀疏疏的写了两个星期的华为2017精英挑战赛的代码，是一个网络中的经典选址问题，这里不做过多的描述，强撸了七八百行代码，这个算是为数不多自己写完的，一直不太知道怎么开始写，然后硬着头皮开始这个事情，居然还是把答案写出来了，不过优化真心做的不够好

这里也就发现一个很大的问题，既有的逻辑中，主要还是面向过程的思维，整个的代码基本上是一个点一个点的解决，这样子造成了代码的不流畅性，于是在这一个星期，找到了浙大翁老师的c++的课程，把面向对象的三大特点以及虚构析构又学习了一遍，温故知新，我想说的就是这个道理。

另外，这本《Thinkging in c++》看豆瓣评分还不错，希望这本书能帮助我解决一些问题，当然这里会持续不停更新这本书的收获：

2.你解决问题的复杂度依赖于你对问题的抽象

The programmer must establish the association
between the machine model
(in the “solution space,” which is the place where you’re modeling that problem, such as a computer)
and the model of the problem that is actually being solved
(in the “problem space,” which is the place where the problem exists)

oop的思想： 从问题空间抽象

The object-oriented approach goes a step farther by providing tools for the programmer to represent elements in the problem space. This representation is general enough that the programmer is not constrained to any particular type of problem.

将 problem space中的elements 称之为 “objects”

通过这种描述，你可以在解决问题的时候，你也可以了解问题本身

Each object looks quite a bit like a little computer; it has a state, and it has operations that you can as

it to perform. However, this doesn’t seem like such a bad analogy to objects in the real world; they all have characteristics and behaviors.
